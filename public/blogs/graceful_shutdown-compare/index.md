> 这篇文章尽量用比较直白的方式聊清楚一个问题：  服务下线的时候，怎样才能尽量不影响正在访问它的请求？

很多人一听“优雅下线”，第一反应可能是：

> 不就是收到退出信号之后，sleep 几秒再关吗？

其实没这么简单。尤其是在 Dubbo 这种 RPC 框架里，一个服务实例下线，影响的不只是它自己，还会影响所有正在调用它的消费者。

如果处理不好，就可能出现这些情况：

- 服务端已经准备退出了，客户端还在继续调用它；
- 请求已经打到服务端了，但服务端进程突然停了，请求直接失败；
- 注册中心里服务已经删了，但客户端本地缓存还没来得及更新；
- 客户端第一次请求失败后，后面还继续请求同一个坏掉的实例。

所以，优雅下线真正要解决的是一个“流量怎么平滑撤走”的问题。

## 1. 优雅下线到底要做到什么？

我觉得可以拆成两个目标。

第一个目标是：

> 新请求不要再打到正在下线的机器。

比如一个 Provider 准备下线了，那么 Consumer 应该尽快知道这件事，把这个 Provider 从自己的可用实例列表里删掉。

第二个目标是：

> 已经打进来的请求，尽量让它正常处理完。

如果一个请求刚进入 Provider，业务逻辑还没执行完，这时候进程直接退出，就会导致调用方拿到失败或者超时。

所以比较理想的下线过程应该是这样：

```text
告诉别人：我准备下线了，别再调我
等待一小会儿：让客户端把我从列表里摘掉
处理完手上的请求
拒绝新的请求
释放连接、关闭协议、退出进程
```

听起来很顺，但真正做起来会遇到一个核心问题：

> “告诉别人我下线了”这件事，并不是瞬间完成的。

## 2. 只靠注册中心够不够？

在 Dubbo 里，最常见的下线方式是通过注册中心。

Provider 下线时，会先从注册中心里把自己注销掉。Consumer 监听注册中心变化，收到通知后，更新本地服务列表，以后就不会再调这个 Provider。

这个思路没有问题，也是最基础、最重要的一条路径。

但它有一个现实问题：注册中心通知是异步的。

也就是说：

```text
Provider 已经从注册中心注销
        ↓
注册中心正在通知 Consumer
        ↓
Consumer 收到通知
        ↓
Consumer 更新本地缓存
        ↓
Consumer 不再调用这个 Provider
```

这中间是有时间差的。

如果 Provider 注销之后马上退出，Consumer 可能还没来得及更新本地缓存，于是请求还是会打过来。这个时候就会出现调用失败。

所以传统方案里通常会有一个等待时间：

```text
先注销注册中心
再等几秒
然后关闭服务
```

这个等待时间就是为了给 Consumer 一点反应时间。

但问题又来了：这个时间该设置多久？

- 设置短了，Consumer 还没更新完，容易出错；
- 设置长了，每次发布、重启都会变慢；
- 小集群可能 1 秒就够了，大集群可能几秒都不稳；
- 不同注册中心、不同网络环境，表现也不一样。

所以，只靠注册中心并不是不行，而是不够稳。

## 3. dubbo-java 的思路：按顺序把服务关干净

dubbo-java 的优雅停机流程比较经典：

```text
收到停机信号
  -> 从注册中心注销 Provider
  -> 等待 Consumer 感知变化
  -> 关闭协议层
  -> 等待在途请求完成
  -> 释放资源并退出
```

这套方案的优点是主线清晰、容易理解，也比较符合传统 RPC 框架的设计方式。

它主要做了三件事：

1. **先从注册中心摘除自己**，让 Consumer 后续不要再路由过来；
2. **等待一段时间**，给注册中心通知和 Consumer 本地缓存刷新留出空间；
3. **等待在途请求完成**，避免进程直接退出导致请求失败。

另外，dubbo-java 还需要处理 Java/Spring 场景下的生命周期问题。比如 Spring 容器关闭、Bean 销毁、Dubbo shutdown hook 的执行顺序，都可能影响优雅停机效果。所以它在 Spring 生态里的适配会更复杂，也更工程化。

但它的不足也比较明显：

> 它比较依赖注册中心通知能够及时到达 Consumer。

如果注册中心通知慢，或者 Consumer 本地缓存更新慢，那么等待时间结束后，仍然可能有请求打到正在关闭的 Provider。

所以 dubbo-java 的方案可以理解成：

> 我先从注册中心下线，然后等一等，希望大家都知道了。

这个方案不差，而且在很多稳定场景下已经够用。只是它对注册中心链路的依赖比较强。

## 4. dubbo-go 的思路：不只等注册中心，而是让 Consumer 更快知道

dubbo-go 的思路更进一步。

它不是否定注册中心，而是认为：

> 注册中心必须有，但不能只靠注册中心。

因为真实环境里，注册中心通知可能延迟，Consumer 本地缓存可能没刷新，长连接也可能还没断。所以 dubbo-go 在注册中心之外，又加了几条感知路径：

```text
Provider 进入 closing 状态
  -> 从注册中心注销
  -> 主动通知已连接的 Consumer
  -> 响应里携带 closing 标记
  -> Consumer 收到后本地摘除该 Provider
  -> 等待在途请求排空
  -> 关闭协议和资源
```

它的核心变化是：

> 下线不再只是 Provider 自己退出，而是让 Consumer 也参与进来，尽快停止把流量打过来。

可以把 dubbo-go 的方案理解成多层保险：

```text
第一层：注册中心通知
第二层：长连接主动通知
第三层：响应里带 closing 标记
第四层：调用失败后识别连接关闭错误
```

任何一层生效，都能减少 Consumer 继续访问下线 Provider 的概率。

## 5. closing 状态：我还活着，但我准备走了

dubbo-go 的一个关键设计是：Provider 收到下线信号后，不是马上关闭，而是先进入 `closing` 状态。

这个状态的含义是：

> 我现在还能处理请求，但已经不应该再被当成正常可用节点了。

进入 closing 后，Provider 可以做几件事：

- 响应里带上 closing 标记；
- 如果协议支持，主动通知 Consumer；
- 等待 Consumer 从本地路由里摘除自己；
- 等待当前请求处理完；
- 最后再关闭协议和连接。

这样下线就不是一个突然动作，而是一个分阶段的过程。

## 6. 主动通知和被动通知：让 Consumer 尽快感知

dubbo-go 主要通过两种方式让 Consumer 更快知道 Provider 要下线。

第一种是**主动通知**。

对于 gRPC、Triple 这类长连接协议，Provider 和 Consumer 本来就有连接。Provider 准备下线时，可以通过健康检查等机制告诉 Consumer：

```text
我快不可用了，不要再往我这里发请求。
```

这样 Consumer 不需要完全等注册中心通知，就能更早摘除这个 Provider。

第二种是**被动通知**。

如果 Consumer 还没感知到下线，仍然发了一个请求过来，Provider 可以正常处理这次请求，并在响应里带上：

```text
closing = true
```

Consumer 收到后就知道：

> 这个 Provider 已经在下线了，下次不要再调它。

这个设计很实用，因为它不会额外增加一次网络调用，而是复用了正常 RPC 响应。

## 7. Consumer 侧摘除：知道还不够，必须真的别再调

仅仅让 Consumer “知道” Provider 要下线还不够。

真正关键的是：

> Consumer 后续路由时，不能再选这个 Provider。

所以 dubbo-go 在 Consumer 侧会把对应 invoker 标记为 closing，并从本地可用列表里摘除。后续负载均衡选择实例时，会跳过这个 Provider。

这点很重要。

因为优雅下线不是服务端一个人的事情。流量是 Consumer 发出来的，如果 Consumer 不改变路由结果，Provider 再怎么声明自己要下线也没用。

## 8. tombstone 和滑动窗口：处理两个细节问题

dubbo-go 里还有两个比较值得看的细节。

第一个是 **tombstone**。

Consumer 收到 closing 后，会把 Provider 从本地列表里删掉。但如果注册中心的旧数据又推过来，可能会把这个 Provider 重新加回来。

所以 tombstone 可以理解成一个短期黑名单：

> 这个实例刚刚因为 closing 被删掉，短时间内不要再把它加回来。

它解决的是旧数据回填的问题。

第二个是**滑动窗口**。

优雅下线时通常会统计当前正在处理的请求数，比如：

```text
activeCount == 0
```

但 counter 为 0 不一定代表流量真的停了。高并发下可能只是某一瞬间刚好没有请求，下一毫秒又来了新请求。

所以 dubbo-go 不只看 active count，还会看最近一段时间内有没有新请求进来。

也就是同时满足：

```text
当前没有在途请求
最近一小段时间也没有新请求
```

这样才能更稳地判断流量已经安静下来。

## 9. 两套方案最大的区别

如果用一句话概括：

> dubbo-java 更像是“按正确顺序把服务关掉”；  
> dubbo-go 更像是“想办法让调用方尽快别再调我”。

可以简单对比一下：

| 方面          | dubbo-java           | dubbo-go                               |
| ------------- | -------------------- | -------------------------------------- |
| 核心思路      | 注册中心摘除 + 等待  | 注册中心 + 多条 Consumer 感知路径      |
| Consumer 感知 | 主要依赖注册中心通知 | 注册中心、主动通知、响应标记、错误识别 |
| 请求排空      | 等待在途请求完成     | active counter + 最近请求时间窗口      |
| 本地摘除      | 等 Directory 刷新    | 收到 closing 后可主动摘除              |
| 复杂度        | 较低，主线清晰       | 较高，但抗异常能力更强                 |
| 更适合        | 传统稳定场景         | 云原生、长连接、滚动发布频繁的场景     |

所以这不是谁先进谁落后的问题，而是侧重点不同。

**dubbo-java 的方案更像一个稳定基线**，简单、清晰、容易维护，也更重视 Java/Spring 生态下的关闭流程。

**dubbo-go 的方案更偏云原生和多协议场景**。它承认注册中心通知可能不及时，所以通过主动通知、响应标记、本地摘除、tombstone、滑动窗口等机制，让流量更快从下线实例上撤走。

最后可以总结成一句话：

> 优雅下线不是 sleep 几秒，也不是简单注销注册中心，而是在一个分布式调用链路里，尽量平滑地把流量从即将退出的机器上挪走。
