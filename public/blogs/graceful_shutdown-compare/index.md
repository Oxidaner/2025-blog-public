> 这篇文章尽量用比较直白的方式聊清楚一个问题：  
> 服务下线的时候，怎样才能尽量不影响正在访问它的请求？

很多人一听“优雅下线”，第一反应可能是：

> 不就是收到退出信号之后，sleep 几秒再关吗？

其实没这么简单。尤其是在 Dubbo 这种 RPC 框架里，一个服务实例下线，影响的不只是它自己，还会影响所有正在调用它的消费者。

如果处理不好，就可能出现这些情况：

- 服务端已经准备退出了，客户端还在继续调用它；
- 请求已经打到服务端了，但服务端进程突然停了，请求直接失败；
- 注册中心里服务已经删了，但客户端本地缓存还没来得及更新；
- 客户端第一次请求失败后，后面还继续请求同一个坏掉的实例。

所以，优雅下线真正要解决的是一个“流量怎么平滑撤走”的问题。

## 1. 优雅下线到底要做到什么？

我觉得可以拆成两个目标。

第一个目标是：

> 新请求不要再打到正在下线的机器。

比如一个 Provider 准备下线了，那么 Consumer 应该尽快知道这件事，把这个 Provider 从自己的可用实例列表里删掉。

第二个目标是：

> 已经打进来的请求，尽量让它正常处理完。

如果一个请求刚进入 Provider，业务逻辑还没执行完，这时候进程直接退出，就会导致调用方拿到失败或者超时。

所以比较理想的下线过程应该是这样：

```text
告诉别人：我准备下线了，别再调我
等待一小会儿：让客户端把我从列表里摘掉
处理完手上的请求
拒绝新的请求
释放连接、关闭协议、退出进程
```

听起来很顺，但真正做起来会遇到一个核心问题：

> “告诉别人我下线了”这件事，并不是瞬间完成的。

## 2. 只靠注册中心够不够？

在 Dubbo 里，最常见的下线方式是通过注册中心。

Provider 下线时，会先从注册中心里把自己注销掉。Consumer 监听注册中心变化，收到通知后，更新本地服务列表，以后就不会再调这个 Provider。

这个思路没有问题，也是最基础、最重要的一条路径。

但它有一个现实问题：注册中心通知是异步的。

也就是说：

```text
Provider 已经从注册中心注销
        ↓
注册中心正在通知 Consumer
        ↓
Consumer 收到通知
        ↓
Consumer 更新本地缓存
        ↓
Consumer 不再调用这个 Provider
```

这中间是有时间差的。

如果 Provider 注销之后马上退出，Consumer 可能还没来得及更新本地缓存，于是请求还是会打过来。这个时候就会出现调用失败。

所以传统方案里通常会有一个等待时间：

```text
先注销注册中心
再等几秒
然后关闭服务
```

这个等待时间就是为了给 Consumer 一点反应时间。

但问题又来了：

这个时间该设置多久？

- 设置短了，Consumer 还没更新完，容易出错；
- 设置长了，每次发布、重启都会变慢；
- 小集群可能 1 秒就够了，大集群可能几秒都不稳；
- 不同注册中心、不同网络环境，表现也不一样。

所以，只靠注册中心并不是不行，而是不够稳。

## 3. dubbo-java 的思路：经典、稳，但更依赖注册中心

dubbo-java 的优雅停机方案可以理解成一条很经典的主线：

```text
收到停机信号
  -> 从注册中心注销 Provider
  -> 等待一段时间，让 Consumer 收到通知
  -> 关闭协议层
  -> 等待正在处理的请求完成
  -> 释放网络资源
```

这套方案的优点是：清晰、稳定、容易理解。

它的核心思想是：

> 注册中心是服务发现的中心，Provider 下线应该先告诉注册中心，再等消费者慢慢收敛。

这很符合传统 RPC 框架的设计方式。

### dubbo-java 做得好的地方

**第一，流程很清楚。**

它没有引入太多额外路径，主要就是注册中心摘除 + 协议关闭 + 等待在途请求。

对于大多数普通场景，这已经能解决很多问题。

**第二，它会关注在途请求。**

也就是说，不是收到停机信号之后马上把进程杀掉，而是尽量等已经进入服务端的请求处理完。

这一点非常重要，因为优雅下线不是“别让新请求进来”就够了，还要照顾已经进来的请求。

**第三，它很好地处理了 Spring 生命周期问题。**

Java 应用经常跑在 Spring 或 Spring Boot 里，而 Spring 自己也有关闭流程。如果 Dubbo 的关闭流程和 Spring 的关闭流程互相抢，就可能出现 Bean 已经销毁了，但 Dubbo 还想用它做清理工作的情况。

所以 dubbo-java 后来对 Spring 场景做了专门处理：尽量让 Dubbo 的关闭动作和 Spring 容器关闭动作对齐。

这点其实很工程化，也很重要。

### dubbo-java 的不足

它最大的问题是：

> 比较依赖注册中心通知能及时到达 Consumer。

如果注册中心通知慢，或者 Consumer 本地缓存更新慢，那么等待时间结束后，仍然可能有请求打到正在关闭的 Provider。

所以 dubbo-java 的传统方案更像是：

> 我先告诉注册中心，然后等一等，希望大家都知道了。

这不是错误，只是它对注册中心通知链路的依赖比较强。

## 4. dubbo-go 的思路：不只等注册中心，而是多给几条路

dubbo-go 的设计思路更进一步。

它不是说注册中心没用，而是认为：

> 注册中心还是必须有，但不能只靠注册中心。

因为在真实生产环境里，注册中心通知可能延迟，Consumer 本地缓存可能还没刷新，长连接状态也可能还没变化。

所以 dubbo-go 做了一套更“保险”的方案：

```text
Provider 准备下线
  -> 标记自己进入 closing 状态
  -> 从注册中心注销
  -> 主动通知部分 Consumer：我准备下线了
  -> 如果还有请求打进来，就在响应里带上 closing 标记
  -> Consumer 收到 closing 后，本地直接摘掉这个 Provider
  -> 等待在途请求处理完
  -> 关闭协议和资源
```

这套方案的核心不是“替代注册中心”，而是：

> 在注册中心通知之外，再加几条更快的感知路径。

可以把它理解成多层保险：

```text
第一层：注册中心通知
第二层：长连接主动通知
第三层：响应里带 closing 标记
第四层：调用失败时识别连接关闭错误
```

任何一层生效，都可以让 Consumer 更快停止访问正在下线的 Provider。

## 5. Provider 先标记 closing：我还没死，但我准备走了

dubbo-go 的一个关键点是：Provider 收到下线信号后，会先进入 closing 状态。

这个状态很好理解：

> 我现在还能处理请求，但我已经准备下线了，请不要再把我当成长期可用的节点。

这个状态不是马上关闭服务，而是告诉框架：

- 接下来响应里要带上 closing 标记；
- 如果协议支持，可以主动告诉 Consumer；
- Consumer 收到后要从本地列表摘掉这个实例；
- 等待一段时间后，再开始拒绝新请求；
- 最后再关闭协议和连接。

这样做的好处是：下线不再是一个突然动作，而是一个有阶段的过程。

## 6. 主动通知：能直接告诉客户端，就不要只等注册中心

对于 gRPC、Triple 这类长连接协议，Provider 和 Consumer 之间本来就维持着连接。

既然已经有连接，那 Provider 准备下线时，就可以尽量主动告诉 Consumer：

> 我这个服务快不可用了，你别再往我这里发请求了。

比如通过健康检查状态，把服务状态从 `SERVING` 改成 `NOT_SERVING`。

Consumer 如果监听了这个健康状态，就能更快知道 Provider 要下线，然后把它从本地可用列表里摘掉。

这个设计的好处是很明显的：

- 不需要等注册中心通知；
- 对已经建立长连接的 Consumer 很快；
- 特别适合 gRPC、Triple 这种协议；
- 注册中心变成兜底，而不是唯一通知方式。

当然，它也不是万能的。

如果 Consumer 没有建立长连接，或者没有监听健康状态，那它就收不到这个主动通知。所以主动通知只能算一条快速路径，不能替代注册中心。

## 7. 被动通知：请求来了也别浪费，顺手告诉它我要下线

主动通知只能覆盖一部分 Consumer。

那其他 Consumer 怎么办？

dubbo-go 还有一个很巧的设计：在响应里带 closing 标记。

假设 Provider 已经准备下线了，但 Consumer 还不知道，仍然发了一个请求过来。

这时候 Provider 不一定马上拒绝它，而是可以正常处理请求，并在响应里顺手带一个标记：

```text
closing = true
```

Consumer 收到这个响应之后，就知道：

> 这个 Provider 已经在下线了，下次不要再调它。

这个方案很实用，因为它没有额外增加一次网络调用，而是复用了正常的 RPC 响应。

它特别适合这种情况：

- Consumer 还没收到注册中心通知；
- 但它刚好还有请求打到这个 Provider；
- Provider 借这次响应把下线状态告诉它；
- Consumer 之后就不再路由到这个 Provider。

这就像你去一家店买东西，店员正常服务完你，然后告诉你：“我们马上关门了，下次别来了。”

它不会打断当前这次服务，但会阻止后续请求继续进来。

## 8. Consumer 收到 closing 后，必须真的摘掉实例

这里只知道 Provider 要下线还不够。

如果 Consumer 只是收到一个 closing 标记，然后打个日志，那没有任何意义。

真正关键的是：

> Consumer 后续路由时，不能再选这个 Provider。

所以 dubbo-go 在 Consumer 侧会做几件事：

1. 把这个 invoker 标记为 closing；
2. 如果 invoker 支持可用性设置，就设置成 unavailable；
3. 后续请求选择实例时，跳过这个 closing invoker；
4. 通知本地 Directory，把这个实例从缓存里移除。

这样 closing 才真正影响了路由结果。

这个地方是 dubbo-go 方案很重要的一点：它不是只在服务端做优雅下线，而是让 Consumer 也参与进来。

因为流量是 Consumer 发出来的。Provider 只说“我要下线了”还不够，Consumer 必须真的“别再发了”。

## 9. tombstone：防止刚删掉的实例又被旧数据加回来

这里还有一个隐藏问题。

Consumer 本地把 closing Provider 删掉之后，注册中心可能还没完全收敛。某些旧的注册中心事件或者本地刷新逻辑，可能又把这个 Provider 加回来了。

这就很尴尬：

```text
Consumer 收到 closing，删掉 Provider
        ↓
旧的注册中心数据又来了
        ↓
Consumer 又把 Provider 加回列表
        ↓
请求又打到正在下线的 Provider
```

为了解决这个问题，dubbo-go 引入了一个类似“短期黑名单”的机制，可以理解成 tombstone。

它的意思是：

> 这个实例我刚刚因为 closing 删掉了，短时间内不要再把它加回来。

这个 tombstone 不会永久存在，而是有一个过期时间。

这样做是为了平衡两个问题：

- 太短：旧数据可能又把实例加回来；
- 太长：如果判断错了，可能影响正常实例恢复。

所以 tombstone 的作用不是替代注册中心，而是保护 Consumer 本地的快速摘除结果，避免被旧数据立刻覆盖。

## 10. 滑动窗口：counter 为 0 不代表真的没人来了

优雅下线里经常会有一个计数器，用来记录当前有多少请求正在处理。

最直觉的写法是：

```text
只要 activeCount == 0，就认为请求处理完了
```

但这个判断其实有问题。

想象一个高并发场景：

```text
10:00:00.000 有请求 A、B、C 正在处理
10:00:00.010 A、B、C 都返回了，activeCount 变成 0
10:00:00.011 新请求 D 又进来了
```

如果框架刚好在 `activeCount == 0` 的瞬间决定关闭，就可能误以为“请求已经排空”，但实际上流量还没停。

所以 dubbo-go 不只是看 counter，还会记录“最近一次收到请求的时间”。

只有同时满足两个条件，才认为可以继续关闭：

```text
当前没有正在处理的请求
并且最近一小段时间内没有新请求进来
```

这就是滑动窗口的意义。

它解决的不是“怎么数请求”，而是：

> 怎么判断流量真的安静下来了，而不是刚好某一瞬间 counter 为 0。

这个设计比单纯看 counter 更稳。

## 11. 连接错误识别：第一次失败后，别一直失败

即使做了注册中心通知、主动通知、响应 closing 标记，仍然可能有漏网之鱼。

比如：

- Consumer 没有收到主动通知；
- 请求还没拿到响应，连接就断了；
- 注册中心通知延迟；
- Provider 已经开始关闭底层连接。

这时候 Consumer 的一次调用可能失败。

但失败之后，Consumer 应该意识到：

> 这个 Provider 可能已经在下线了，后面不要再一直调它。

所以 dubbo-go 会识别一些连接关闭相关错误，比如：

- 连接正在关闭；
- client 已关闭；
- invoker 已销毁；
- gRPC 返回 unavailable；
- gRPC 返回 canceled。

如果判断这是下线相关错误，就把这个 Provider 标记为 closing，并从本地路由里移除。

这个设计不能避免第一次失败，但可以减少连续失败。

## 12. 两套方案最大的区别

如果用一句话概括：

> dubbo-java 更像是“先通知注册中心，然后等待大家知道”；  
> dubbo-go 更像是“注册中心通知照做，但我还要通过多种方式让 Consumer 更快知道”。

再简单一点说：

```text
dubbo-java：注册中心是主路径
dubbo-go：注册中心是主路径之一，但不是唯一依赖
```

可以用这个表来理解：

| 方面          | dubbo-java             | dubbo-go                                      |
| ------------- | ---------------------- | --------------------------------------------- |
| 核心思路      | 依赖注册中心通知消费者 | 注册中心 + 主动通知 + 被动通知 + 错误感知     |
| 下线等待      | 反注册后等待一段时间   | 等待期间还会尽量把 closing 状态传给 Consumer  |
| 请求排空      | 等在途请求完成         | active counter + 最近请求时间窗口             |
| Consumer 感知 | 主要靠注册中心刷新     | 可以通过 health、attachment、错误识别更快感知 |
| 本地摘除      | 等 Directory 正常刷新  | 收到 closing 后可直接摘除本地实例             |
| 复杂度        | 较低，主线清晰         | 较高，但更抗异常场景                          |
| 更适合        | 传统稳定场景           | 云原生、长连接、注册中心通知可能延迟的场景    |

## 13. dubbo-java 好在哪里？

dubbo-java 的方案并不落后，它的优点很明显。

### 13.1 简单、清晰、容易维护

它的主流程非常容易理解：

```text
注销注册中心
等待消费者更新
关闭协议
等待请求完成
释放资源
```

这类设计通常更容易维护，也更不容易引入复杂状态问题。

### 13.2 和 Java 生态结合得更深

Java 应用大量使用 Spring。优雅停机不仅是 Dubbo 自己的事情，还要考虑 Spring 容器什么时候关闭、Bean 什么时候销毁、shutdown hook 谁先执行。

dubbo-java 在这块处理得比较成熟。

### 13.3 更像一个稳定基线

对于很多业务来说，只要注册中心比较稳定，等待时间设置合理，dubbo-java 这种方案已经够用。

它不是最激进的方案，但很稳。

## 14. dubbo-java 不足在哪里？

主要不足也很清楚：

### 14.1 对注册中心通知比较依赖

如果注册中心通知慢，Consumer 本地缓存更新慢，Provider 等待结束后仍可能被请求到。

### 14.2 等待时间不好调

这个时间短了不安全，长了影响发布速度。

而且它和业务规模、注册中心类型、网络情况都有关系，很难一个默认值适配所有场景。

### 14.3 对“Consumer 快速感知”做得不够多

传统方案更多是等待 Consumer 通过注册中心感知，而不是通过协议、响应、错误等多条路径主动加速感知。

## 15. dubbo-go 好在哪里？

dubbo-go 的优点是更贴近真实生产中的复杂情况。

### 15.1 不把注册中心当成唯一可靠路径

它承认注册中心通知可能有延迟，所以加了其他路径。

这个思路很工程化：

> 不假设某一个组件永远及时，而是用多条路径降低整体失败概率。

### 15.2 Consumer 侧真的参与了优雅下线

Provider 自己关得优雅还不够，Consumer 不再发请求才是关键。

dubbo-go 让 Consumer 在收到 closing 信号后，直接影响本地路由结果，这个设计比较完整。

### 15.3 滑动窗口很实用

单纯 counter 为 0 不一定代表流量停止。滑动窗口让“请求排空”的判断更稳，尤其适合下线期间还允许短暂接收请求的场景。

### 15.4 tombstone 解决了一个很真实的问题

很多人会想到“收到 closing 后删掉实例”，但容易忽略“旧注册中心数据又把实例加回来”的问题。

tombstone 这个设计就是为了解决这个细节。

## 16. dubbo-go 不足在哪里？

dubbo-go 的问题也很明显：复杂。

它涉及：

- shutdown 主流程；
- Provider filter；
- Consumer filter；
- Directory；
- 协议 callback；
- health watch；
- tombstone；
- 多个超时时间和 TTL。

这就带来几个问题。

### 16.1 理解成本更高

读代码时不能只看一个文件，而要串起多个模块。

### 16.2 参数更难调

比如：

- 主动通知等多久？
- sliding window 多长？
- closing invoker 保留多久？
- tombstone 多久过期？

这些都没有绝对标准，要根据实际部署环境调。

### 16.3 某些错误识别仍然偏经验化

比如通过错误字符串判断连接是否正在关闭，这种方式有一定脆弱性。

更理想的方式是协议层返回更明确的结构化错误。

### 16.4 测试要求更高

这种多模块协作的设计，单元测试不够，还需要端到端测试。

比如要模拟：

- 注册中心通知延迟；
- 长连接断开；
- Provider 正在处理慢请求；
- Consumer 本地缓存没刷新；
- 旧数据把实例重新加回来；
- 多个 Consumer 同时调用。

否则很难证明它真的减少了失败请求。

## 17. 为什么 Go 和 Java 的设计不一样？

这不是谁先进谁落后的问题，而是背景不同。

### 17.1 Java 更关注 Spring 生命周期

Java 应用经常和 Spring 深度绑定，所以优雅停机必须处理 Spring 容器关闭、Bean 销毁、JVM hook 顺序这些问题。

### 17.2 Go 更贴近云原生进程模型

Go 服务更常见的模型是：

```text
一个进程
收到 SIGTERM
在 K8S 给定的时间内退出
```

所以它更自然地关注：

- SIGTERM 后怎么排空请求；
- 长连接 Consumer 怎么感知；
- Provider 退出前怎么减少失败请求；
- 本地缓存怎么快速摘除。

### 17.3 gRPC/Triple 这类协议给了更多通知方式

长连接协议本身就适合做健康状态监听。

既然协议层能告诉 Consumer “我不可服务了”，那就没必要完全等注册中心。

这也是 dubbo-go 方案能做得更主动的原因之一。

## 18. 我觉得最值得总结的设计思想

这套 dubbo-go 方案最值得总结的不是某个代码细节，而是一个思路：

> 优雅下线不是服务端自己退出得慢一点，而是让整个调用链路里的流量逐步收敛。

它把这个问题拆成了几层：

```text
注册中心：负责最终一致性
协议层：负责快速通知
Provider filter：负责请求计数和响应标记
Consumer filter：负责识别 closing 并停止调用
Directory：负责本地实例摘除
tombstone：负责防止旧数据回填
```

这样看，优雅下线就不再是一个简单的 shutdown hook，而是一个分布式系统里的流量治理问题。

## 19. 可以怎么继续优化？

如果后续继续优化，我觉得可以从几个方向做。

### 19.1 错误类型更结构化

现在有些连接关闭错误需要靠字符串识别。更好的方式是定义明确的错误类型，比如：

```text
ProviderClosingError
ClientClosedError
InvokerDestroyedError
TransportClosingError
```

这样 Consumer 判断时会更准确。

### 19.2 增加更多可观测指标

优雅下线到底有没有效果，需要数据说明。

可以加一些指标：

- 主动通知成功次数；
- 主动通知失败次数；
- Consumer 通过 attachment 感知 closing 的次数；
- tombstone 命中次数；
- Provider 请求排空耗时；
- shutdown 总耗时；
- 因 closing 被跳过的 invoker 次数。

有了这些指标，线上调参会容易很多。

### 19.3 给出推荐参数

比如：

- 总 shutdown timeout 要小于 K8S 的 termination grace period；
- notify timeout 不要占太多总时间；
- sliding window 不要太长；
- tombstone TTL 要覆盖注册中心常见延迟；
- closing invoker TTL 要和注册中心刷新周期匹配。

否则功能虽然有了，但用户不知道怎么配。

### 19.4 做端到端压测

优雅下线最终要看的是发布过程中失败请求有没有下降。

所以最好用真实压测场景验证，比如：

```text
持续压测
同时滚动重启 Provider
观察 Consumer 失败请求数量
对比开启/关闭增强下线能力的差异
```

这个结果会比单纯讲设计更有说服力。

## 20. 总结

dubbo-java 和 dubbo-go 的优雅下线设计，其实代表了两种不同风格。

dubbo-java 的方案更经典：

> 先从注册中心摘除，再等待消费者更新，最后关闭协议和资源。

它简单、清晰、稳定，也很好地处理了 Java/Spring 生态下的生命周期问题。

dubbo-go 的方案更偏云原生和多协议场景：

> 注册中心仍然保留，但不只依赖注册中心；通过主动通知、响应标记、错误识别、本地摘除和 tombstone，让 Consumer 更快停止访问正在下线的 Provider。

它复杂一些，但抗异常能力更强，尤其适合注册中心通知不及时、长连接较多、K8S 滚动发布频繁的场景。

如果用一句话总结：

> dubbo-java 更像是“按正确顺序关掉服务”；  
> dubbo-go 更像是“想办法让调用方尽快别再调我”。

而在真实生产里，后者往往更接近我们遇到的问题。

优雅下线不是 sleep 几秒，也不是简单地注销注册中心。它真正要解决的是：在一个分布式调用链路里，如何尽量平滑地把流量从一台即将退出的机器上挪走。
